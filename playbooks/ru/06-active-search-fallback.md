# Плейбук 06 — Active Search Fallback (запасной активный поиск)

> **Когда запускать:** ТОЛЬКО как запасной вариант, когда inbound недостаточен или по прямому запросу пользователя.
> **Приоритет:** НИЗКИЙ. Это **второй** цикл, не основной. См. `BOOTSTRAP.md` §2–3.
> **Язык:** RU/EN под роль. **Режим:** human-in-the-loop, **качество важнее количества**.

---

## Цель

Точечные, высококачественные отклики — а не массовая рассылка (масса убивает inbound-сигнал). Каждое приложение проходит alignment-шлюз.

## Принципы (соблюдать строго)

1. **Качество > количество.** Лучше 2 сильных отклика, чем 20 шаблонных.
2. **Капка объёма:** не больше **3 откликов в неделю** (по умолчанию; пользователь может изменить).
3. **Alignment-шлюз обязателен** — каждая роль проходит оценку плейбука 02. Без зелёного/жёлтого — не откликаться.
4. **Tailored, не generic.** Резюме/сопроводительное адаптируются под конкретную роль (упор на §11 кейсы).
5. **Никогда не применять автоматически.** Только черновик — пользователь отправляет.

## Выход

`output/active-search-YYYY-MM-DD.md` (RU):
```md
# Активный поиск (fallback) — неделя YYYY-MM-DD
## Контекст
Почему сейчас активный поиск (запасной)
## Целевые роли (из watchlist 01, прошедшие alignment 02)
## Отклики — черновики
### [Компания] — [роль] — alignment 🟢/🟡
- Почему созвучно
- Tailored-выжимка из CV (акцент на релевантные кейсы §11)
- Сопроводительное (RU или EN под роль)
## Капка: X/3 использовано
```

## Правила

1. Брать роли **только** из watchlist плейбука 01, прошедших alignment плейбука 02.
2. Каждое приложение — индивидуальное. Никаких шаблонных mass-apply.
3. Если за неделю нет созвучных ролей — честно «нет подходящих», **не понижать планку**.
4. Подсветить пользователю: активный поиск — запасной; основной фокус остаётся на inbound (плейбуки 03–04).
5. Логировать результат (ответ/отказ/тишина) — это обратная связь для позиционирования.
